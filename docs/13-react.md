# 13 — ReAct (Reason + Act)

**Definition.** ReAct is a prompting paradigm in which an LLM alternates natural-language *reasoning traces* ("Thought: …") with *actions* on external tools ("Action: search(...)") and *observations* ("Observation: …"). The loop continues until the model emits a final answer. Introduced in Yao et al. (ICLR 2023), it is the pattern most modern agent harnesses still descend from.

## Problem it solves

Before ReAct, two paradigms competed. **Chain-of-Thought** produced rich reasoning but couldn't interact with the world — the model confabulated facts rather than looking them up. **Tool-calling** produced actions but without visible reasoning — the model's choices were opaque and brittle. ReAct fuses them: each action is preceded by a thought that justifies it, and each observation is interpreted by a thought before the next action. The result: more grounded reasoning, more purposeful actions, fewer hallucinations, and transcripts a human can debug.

## Mechanism

A ReAct trajectory is a sequence of triples, plus a final answer:

```
Thought₁ → Action₁ → Observation₁
Thought₂ → Action₂ → Observation₂
...
Thought_n → Answer
```

The prompt shows the model a few-shot demonstration of the format, then the current task. On each turn the model generates the next Thought and Action; the harness parses the Action, executes it, and appends the Observation for the next turn.

Crucially:

- **Thoughts are generated by the model, not the harness.** They are part of the token stream — no separate "planning" call.
- **Actions have a small grammar** (`search[query]`, `lookup[term]`, `finish[answer]` in the original paper; arbitrary tool calls in modern implementations).
- **The loop terminates** when the model emits `finish[...]` (or, in tool-call-schema harnesses, when it stops requesting tools).

Modern harnesses replace the original text grammar with structured tool-call schemas (Claude's tool use, OpenAI function calling), but the Thought/Action/Observation shape survives — "Thought" becomes the reasoning that precedes the tool call, often rendered as natural prose in the model's output.

## Concrete pattern

Canonical few-shot snippet (from HotpotQA-style tasks):

```
Question: Which magazine was started first, Arthur's Magazine or First for Women?

Thought 1: I need to find the founding date of each.
Action 1: search[Arthur's Magazine]
Observation 1: Arthur's Magazine (1844-1846) was an American literary periodical ...

Thought 2: Arthur's was 1844. Now First for Women.
Action 2: search[First for Women]
Observation 2: First for Women is a women's magazine published by A360media since 1989.

Thought 3: Arthur's (1844) predates First for Women (1989).
Action 3: finish[Arthur's Magazine]
```

In a modern tool-calling form:

```
assistant> I need the founding dates. Let me search.
  tool_call: web_search(query="Arthur's Magazine founding year")
tool_result> 1844.
assistant> Now the other one.
  tool_call: web_search(query="First for Women magazine founded")
tool_result> 1989.
assistant> Arthur's Magazine was first. Final answer: Arthur's Magazine.
```

The "thought" is embedded in the natural-language commentary around the tool calls, and the harness still gets the structured benefit of explicit reasoning before each action.

## Variants & related techniques

- **Plan-and-Solve** ([16-plan-and-solve.md](16-plan-and-solve.md)) front-loads all planning before any action. ReAct is strictly interleaved.
- **ReWOO** ([17-rewoo.md](17-rewoo.md)) decouples reasoning from observation, emitting a full action plan first and fetching in parallel. Lower latency; less adaptive.
- **Reflexion** ([14-reflexion.md](14-reflexion.md)) wraps ReAct with an end-of-trajectory reflection.
- **Tree of Thoughts / LATS** ([15-tree-of-thoughts-lats.md](15-tree-of-thoughts-lats.md)) explore multiple ReAct trajectories in parallel with a value function to pick winners.
- **Self-Ask** is a sibling where the model explicitly writes "Intermediate question:" before each action — very similar in spirit.

## Failure modes & anti-patterns

- **Reasoning theater.** The model writes confident thoughts that don't actually drive its action choice — post-hoc rationalization. Fix: make thoughts short and actionable; judge by whether the next action would change if the thought were different.
- **Action loops.** The model repeatedly calls the same action with the same arguments, unable to progress. Fix: detect repetition in the harness and inject "you already tried this, choose a different approach".
- **Observation dumping.** Raw tool output (100k tokens of HTML) fills the next Thought context and the model loses the plot. Fix: truncate/summarize observations before feeding them back — the "O" step is where most harness work actually lives.
- **Early termination.** Model emits `finish` with a guess rather than searching further. Fix: verifier loop ([11-verifier-evaluator-loops.md](11-verifier-evaluator-loops.md)) or require evidence citations.
- **Tool-choice overfitting.** The few-shot examples train the model to always start with `search`. Fix: diverse few-shots or switch to zero-shot with rich tool descriptions.
- **Drift in long trajectories.** Thoughts become vaguer as context grows; actions stop connecting to the original question. Fix: persistent task card (see [12-todo-scratchpad-state.md](12-todo-scratchpad-state.md)).

## When to use (and when not to)

**Use** ReAct (or its tool-calling descendants) when:

- The task requires interleaving reasoning with external information (QA, web navigation, coding).
- You want transparent, debuggable trajectories.
- Tool outputs are informative enough to shape subsequent reasoning.

**Don't** use ReAct when:

- You already know the full pipeline of actions — ReWOO or a hard-coded script is faster.
- The task is pure reasoning with no external info — plain chain-of-thought is enough.
- The tools are so cheap and independent that speculative parallel calls beat sequential thought-action.
