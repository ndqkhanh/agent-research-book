## What the paper is about, in one paragraph

Autogenesis argues that most "self-improving agent" research (Voyager-style skill accretion, Reflexion-style trajectory rewrites, Hyperagents-style self-modification) improves behaviour *inside an otherwise fixed substrate*. Deeper self-improvement — rewriting the system prompt, reshaping tool schemas, mutating memory — routinely breaks live systems because the substrate is a monolith with no typed seams. The paper's response is protocol-level: model every mutable thing (prompts, agents, tools, environments, memory) as a first-class, versioned, URI-addressable *resource* with a lifecycle (RSPL), and expose a closed-loop operator interface — propose, assess, commit, rollback — that every self-modification must traverse (SEPL). The Autogenesis System (AGS) is the reference multi-agent runtime that dynamically instantiates, retrieves, and refines those resources during execution. The effect, stated in software-engineering terms, is "git + CI + feature flags applied to the entire agent system, not just to its code."

## Contribution claims (as stated by the paper)

Distilled from the abstract and the introductory framing; quoted where possible.

- "Existing LLM-based agent protocols inadequately address cross-entity lifecycle and context management, version tracking, and evolution-safe update interfaces."
- A **decoupling claim**: "what evolves" (resources) is separated from "how evolution occurs" (protocol). The paper positions this as its primary advance over A2A-style agent-to-agent protocols and MCP-style tool protocols, both of which conflate the two.
- **RSPL (Resource Substrate Protocol Layer):** prompts, agents, tools, environments, and memory are "modeled as protocol-registered resources" with explicit state, lifecycle, and versioned interfaces.
- **SEPL (Self-Evolution Protocol Layer):** a closed-loop operator interface for "proposing, assessing, and committing improvements with auditable lineage and rollback."
- **AGS (Autogenesis System):** a self-evolving multi-agent system that "dynamically instantiates, retrieves, and refines protocol-registered resources during execution."
- An empirical claim: "consistent improvements over strong baselines" on "multiple challenging benchmarks that require long-horizon planning and tool use across heterogeneous resources."

## Method — architecture sketch

### The five resource types (RSPL)

Each is a typed entity with a URI, version, schema, and lifecycle state `draft → active → deprecated → retired`. Changes are patches with lineage, not in-place edits.

| Resource | Examples | What "versioning" buys |
|---|---|---|
| Prompt | system prompt, role prompts, few-shot exemplars | diff/rollback on wording edits |
| Agent | subagent definitions, tool allowlists, system posture | rollback of capability shifts |
| Tool | schemas, executors, permission metadata | schema-migration safety |
| Environment | sandboxes, worktrees, credential scopes | reproducible execution |
| Memory | episodic, semantic, scratchpad stores | schema-level memory migration |

### The self-evolution loop (SEPL)

A four-stage operator interface that every self-modification traverses:

1. **Propose.** A meta-agent (or a sub-loop) proposes a patch against a resource URI, e.g. `prompt://researcher@v3 → @v4`, carrying a rationale.
2. **Assess.** Automated evaluation (benchmark runs, LLM-as-judge, golden traces) scores the proposed version vs. the active baseline, with explicit gating criteria.
3. **Commit.** If gates pass, the new version becomes active atomically; the prior version remains retrievable.
4. **Rollback.** On regression or external signal, revert atomically to a prior version.

This is the same control loop a [verifier-evaluator loop](./11-verifier-evaluator-loops.md) runs over *trajectories*, but applied over *resource versions*. The invariant is: the substrate is always deployable, because every step is gated.

### AGS — the runtime

AGS is the integration layer: a multi-agent orchestrator that consults the resource registry on every turn, so a resource swap during training is immediately visible to subsequent agent instances. The paper's novelty claim here is not the orchestrator itself (several 2024–2025 systems have this) but that instantiation is *protocol-driven*, so adding a new resource type is an API contract rather than a code change.

## Harness-engineering implications

### Against the four pillars we track

In our [four-pillars](./44-four-pillars-harness-engineering.md) framing — *loop, context, tools, safety* — Autogenesis sits primarily in the **tools** and **safety** pillars but has a loop-layer cost:

- **Loop.** SEPL adds a second, slower outer loop around the agent loop: every turn can run through the resource registry, and a subset of turns trigger propose/assess/commit. Practitioners should budget this latency; it is not a free abstraction.
- **Context.** Resources are context. A resource fetch replaces what used to be static prompt concatenation with a registry lookup; this is closer to [memory files](./09-memory-files.md) than to retrieval, and needs its own compaction discipline.
- **Tools.** Tool schemas become versioned artefacts. This is the right primitive and the one most likely to be copied even by teams that reject the rest of the protocol.
- **Safety.** The rollback guarantee is the paper's most operationally valuable claim — it turns self-modification from a "could break prod" event into a reversible transaction.

### Compared to [Hermes Agent (doc 55)](./55-hermes-agent-self-improving.md)

Hermes is a *self-improving agent system* — the improvement signal is folded into the main loop. Autogenesis is a *self-evolving protocol* — improvement is a separate, gated control plane. The two are compatible: a Hermes-style inner improvement loop could emit proposals that Autogenesis's SEPL then assesses and commits. In our corpus, Hermes answers "how do I improve?" and Autogenesis answers "how do I commit the improvement without breaking anything?"

### Compared to [Voyager (doc 19)](./19-voyager-skill-libraries.md)

Voyager's skill library is the prototypical self-evolving resource: new skills are appended, named, and retrieved. But Voyager has exactly *one* resource type (skill code), no rollback, and no schema for the skill interface — in Autogenesis terms, a single-resource, no-SEPL substrate. The generalisation from Voyager to Autogenesis is: five resource types instead of one, and a gated lifecycle instead of an append-only list. A practitioner already running a Voyager-style skill registry gets the 80/20 of Autogenesis by adding versioning, gating, and rollback to that one registry, without adopting the full five-resource protocol on day one.

### Compared to [SemaClaw (doc 54)](./54-semaclaw-general-purpose-agent.md)

SemaClaw is a general-purpose agent with a fixed, well-engineered substrate. Autogenesis does not compete with SemaClaw — it complements it: SemaClaw is what you'd put *behind* an AGS instance when the resources are mature enough that the dominant failure mode is "substrate is static, world is not." The question to ask before adopting Autogenesis on top of a SemaClaw-like base is whether your regression rate from manual substrate edits is high enough to justify the protocol tax.

### The 2026 self-evolving-agent landscape around it

Autogenesis should be read alongside, not in isolation from, the SEA work that has surfaced in the last eight months:

- **AgentEvolver (arXiv:2511.10395).** A system-level take on efficient self-evolution of agents; where Autogenesis says "here's the protocol," AgentEvolver studies "here's how to do the evolution step without burning compute." Complementary.
- **Agent0 (arXiv:2511.16043).** "Self-evolving agents from zero data via tool-integrated reasoning." Pushes the data-free extreme of SEA; Autogenesis is agnostic on where proposals come from, so Agent0-style generation plugs into SEPL as a proposer.
- **Comprehensive SEA Survey (arXiv:2508.07407) and "What, When, How, Where to Evolve" (arXiv:2507.21046).** Taxonomies that place Autogenesis in the "protocolised" quadrant of SEA — explicit interfaces rather than implicit fine-tuning.
- **Experience-driven lifelong-learning benchmark (OpenReview: vznmtmUPmA).** The benchmark that Autogenesis-class systems *ought* to be evaluated on, and that I'd want to see v2 of Autogenesis report against.
- **EvoAgentX (`evoagentx.github.io`).** An open-source SEA runtime; the closest thing to an AGS reference implementation in the wild.

The unifying observation: as of April 2026, SEA is splitting into (i) protocol work (Autogenesis), (ii) efficient-evolution work (AgentEvolver), (iii) data-free evolution (Agent0), and (iv) benchmarks/surveys. Autogenesis is the strongest entrant in (i) but stays out of (ii)–(iii).

## Limitations and open questions

What the paper admits, and what it leaves unstated.

- **No numbers in the abstract.** "Consistent improvements over strong baselines" is the strongest claim surfaced publicly. Any practitioner decision should wait on the full results section; I was unable to extract it cleanly via WebFetch.
- **No baselines named publicly.** "Strong baselines" is unqualified. Is the comparator a single-agent ReAct harness, a full Voyager-style system, or a 2025 production agent? The answer changes the claim's force by an order of magnitude.
- **Evaluation cost of SEPL.** The Assess stage runs benchmark suites, LLM judges, or golden traces on every proposed patch. On a busy AGS, this is an O(proposals × evaluation) cost that the paper's framing elides. A practical deployment needs at least (a) tiered gates (cheap-check before expensive-check) and (b) amortised evaluation across batches of proposals.
- **Who writes the evaluator?** SEPL presupposes a high-fidelity evaluator. If the evaluator is itself learned, it can drift, and the protocol inherits the drift. The paper does not treat the evaluator as a versioned resource, though by its own logic it should be — the evaluator is arguably the sixth RSPL resource type.
- **Proposal quality.** The paper specifies the lifecycle but not the proposer. In practice, a mediocre proposer will produce a flood of regressions that all (correctly) roll back, consuming eval budget without progress. [Reflexion (doc 14)](./14-reflexion.md)-style proposers and Agent0-style proposers are plausible candidates but not studied here.
- **Cross-resource consistency.** A patch to `tool://search@v7` may require a coordinated patch to `prompt://researcher@v5` that references the tool's schema. The paper asserts lineage but does not describe *transactions* across resources. Without cross-resource atomicity, a rollback on one resource can leave its dependents referring to a retired version.
- **Schema evolution of memory.** Migrating a memory store's schema while preserving historical episodes is a known-hard problem that the five-resource taxonomy treats uniformly with prompt edits. Prompts are stateless; memory is not. The uniform treatment is elegant in the abstract and difficult in practice.
- **No attack surface discussion.** A self-evolving substrate is also a *self-modifying* substrate, which is a large attack surface. The paper does not treat [prompt injection (doc 22)](./22-guardrails-prompt-injection.md) or [malicious intermediaries (doc 35)](./35-malicious-intermediary-attacks.md) as threats against SEPL, though a malicious proposal that passes a weak evaluator is exactly the kind of pathway a security-minded practitioner would want guarded.
- **Interoperability.** RSPL defines an interface, but there is no disclosed registry standard (comparable to MCP's schema). Without one, each AGS deployment is its own island.

## What a practitioner should copy vs. leave behind

**Copy.**

- **Versioned prompt and tool resources with URIs.** This is the single highest-leverage takeaway. Even without SEPL, moving from "system prompt is a string literal" to "system prompt is `prompt://<name>@<version>` with a registry" unblocks diff review, A/B testing, and rollback.
- **An atomic rollback primitive.** Every self-modifying agent should have a one-call rollback to the last-known-good substrate. If you cannot implement rollback, you do not have self-modification — you have self-destruction with extra steps.
- **A gated Propose → Assess → Commit loop.** Even a lightweight version (golden-trace check before commit) catches most regressions. This is the agent-substrate analogue of [hooks (doc 05)](./05-hooks.md) gating shell commands.
- **Typed resources with lifecycle states.** `draft → active → deprecated → retired` maps cleanly onto feature-flag patterns teams already run; no invention needed.

**Leave behind, or at least defer.**

- **The full five-resource protocol on day one.** Start with prompts and tools (where diffs are obvious and evaluation is cheap). Environments and memory are harder to version and the payoff is smaller early.
- **A general-purpose proposer.** Until you have a proposer that is notably better than manual edits, the SEPL loop will be a treadmill of regressions. Keep the human-in-the-loop until the propose→commit acceptance rate justifies automation.
- **Assuming the evaluator is free.** Budget the evaluator as a first-class cost. If your evaluator is not itself evaluated, your self-evolving agent is evolving toward your evaluator's blind spots, not toward the task.
- **A full runtime rewrite.** You almost certainly do not need AGS-the-system; you need RSPL-the-discipline. Take the discipline, leave the runtime until the discipline pays for itself.
