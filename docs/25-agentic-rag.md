# 25 — Agentic RAG with Self-Critique

**Definition.** Agentic RAG moves retrieval-augmented generation from a fixed "retrieve → read → respond" pipeline to a dynamic loop in which the agent *decides whether to retrieve, what to retrieve, when to retry, and when enough evidence has been gathered*. Self-critique layers add relevance checks, citation verification, and re-query loops. The result is a retrieval system that behaves like a careful researcher, not a keyword-triggered autocomplete.

## Problem it solves

Classic RAG assumes one query, one retrieval, one response. That works for simple factoid questions and fails on almost everything else:

- **Multi-hop questions** need retrieval chains ("find X's location; then the weather there; then translate").
- **Ambiguous queries** produce poor initial matches; the system should refine, not forge ahead.
- **Mixed-source needs** may require a knowledge base, a web search, and the user's prior messages — each chosen per question.
- **Hallucinated citations** happen when the model uses retrieved docs as atmospheric context rather than actual evidence; outputs look grounded but aren't.

Agentic RAG treats retrieval as a tool the agent plans with, criticizes, and re-invokes — closer to how a researcher actually works.

## Mechanism

Key components:

1. **Retrieval as a tool.** Instead of a fixed pre-pass, `retrieve(query, source, k)` is a normal tool the agent calls in its loop. The agent decides when and how.
2. **Query rewriting.** User questions are rarely retrieval-optimal. A sub-step rewrites into search-friendly queries (sometimes multiple, for different angles). HyDE ("generate a hypothetical answer, embed that") is a common trick.
3. **Router over sources.** Agent picks among a knowledge base, a web search, a SQL tool, a documentation MCP server, etc. Source choice is its own decision.
4. **Relevance self-critique.** For each retrieved chunk, the agent (or a cheaper classifier) judges *is this actually relevant?* Irrelevant results are discarded before synthesis — they don't pollute context.
5. **Re-query on failure.** If nothing relevant came back, rewrite the query and try again rather than hallucinating from weak evidence.
6. **Synthesis with citation binding.** The answer cites specific chunks. A final verifier checks that each citation actually supports its claim (Self-RAG's "IsSup" step, roughly).
7. **Stopping condition.** Stop when evidence suffices *for the question asked* — not when a step budget runs out.

Corrective RAG (CRAG) formalizes the grading step: a lightweight retrieval evaluator scores each document as "Correct", "Ambiguous", or "Incorrect", triggering web-search fallback when knowledge-base retrieval is insufficient.

## Concrete pattern

LangGraph-style agentic RAG flow (condensed):

```python
def agentic_rag(question):
    plan = planner.draft_queries(question)           # e.g., 3 queries
    evidence = []
    for q in plan:
        docs = retriever.search(q, k=6)
        graded = [(d, grade_relevance(q, d)) for d in docs]
        kept = [d for d, g in graded if g == "relevant"]
        if not kept:
            q_rewrite = rewriter.rewrite(q, docs)
            docs = retriever.search(q_rewrite, k=6)
            kept = [d for d in docs if grade_relevance(q_rewrite, d) == "relevant"]
        if not kept:
            kept = web_search(q)                     # fallback
        evidence.extend(kept)

    draft = synthesizer.draft(question, evidence)
    verdict = citation_checker.verify(draft, evidence)
    if not verdict.all_supported:
        draft = synthesizer.refine(draft, evidence, verdict.missing)
    return draft
```

Self-RAG introduces "reflection tokens" — small control signals generated by the model itself — deciding whether to retrieve more, whether a passage is relevant, and whether the answer is supported. The tokens become hooks for the harness to conditionally branch the flow.

## Variants & related techniques

- **Self-RAG** (Asai et al., arXiv:2310.11511) — model-internal reflection tokens steer retrieval and critique.
- **Corrective RAG (CRAG)** — explicit retriever evaluator + web-search fallback.
- **GraphRAG** (Microsoft Research) — retrieval over knowledge graphs rather than chunks; strong on multi-entity reasoning.
- **HyDE** — generate a hypothetical answer; embed *that* instead of the query; retrieve documents similar to the hypothetical answer. Often improves recall on vague queries.
- **Hybrid retrieval** — sparse (BM25) + dense (embeddings) + reranker. Agentic RAG can call any of these.
- **Retrieval planning via ReWOO** ([17-rewoo.md](17-rewoo.md)) — batch retrievals in parallel with placeholders when they're independent.
- **Tool-augmented verifier** (CRITIC, [18-chain-of-verification-self-refine.md](18-chain-of-verification-self-refine.md)) — use real tools to check cited claims, not just another LLM.
- **Chunking and indexing strategy** — hierarchical chunking, sentence-windows, parent-document retrieval. Improvements here help every agentic RAG pattern.

## Failure modes & anti-patterns

- **Retrieval as default.** Agent retrieves for every question, even trivial ones. Cost balloons. Fix: classifier for "does this need retrieval?" before calling retrieve.
- **Over-retrieval.** k=50 everywhere; context bloat; relevant signals drowned. Fix: reranker + cap on final kept chunks; aggressive relevance grading.
- **Citation theater.** Every sentence has a bracket citation, but half don't support the claim. Fix: explicit citation verifier; refuse output if unsupported claims remain.
- **Loop on bad query.** Rewriter produces the same query variants forever. Fix: detect repetition; fall back to web search or ask the user.
- **Mis-routed source.** Agent asks the SQL tool a freeform question and gets an error; tries again instead of switching to the docs tool. Fix: router classifier with confidence thresholds.
- **No freshness awareness.** Retrieved documents are two years stale; the answer is wrong for today. Fix: index with timestamps; prefer newer; for time-sensitive questions, add web search.
- **Treating retrieval output as instructions.** Injected prompts in docs can hijack the agent — see [22-guardrails-prompt-injection.md](22-guardrails-prompt-injection.md).

## When to use (and when not to)

**Use** agentic RAG when:

- Queries are varied, complex, or multi-hop.
- You have multiple sources worth routing among.
- Hallucination is the dominant failure mode and ground truth is retrievable.
- You can afford multiple LLM calls per answer.

**Don't** use it when:

- Questions are narrow and well-served by a fixed retrieve-then-read pipeline.
- You have no reliable relevance grader — the self-critique layer becomes noise.
- Latency budget is tight — a single retrieval + one LLM pass is faster.
- Your knowledge base is small enough to fit in context; skip retrieval entirely.
